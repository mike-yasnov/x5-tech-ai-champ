#### Ветка 1: `feature/advanced-solver-strategies` (PR #2)

**Идея:** Добавить продвинутые стратегии — beam search, послойную укладку, LNS.

**Что сделано:**
- Beam search (ширина луча = K параллельных частичных решений)
- Layer-based packing (послойное заполнение)
- LNS destroy-repair (первый прототип)

**Статус:** Открытый PR, не смёрджен. Код был взят как база для дальнейших веток.

**Вывод:** Каждый из этих подходов по отдельности не дал значимого прироста на бенчмарках. Beam search был слишком медленным для 900 мс бюджета. Но идеи оказались полезны как компоненты в составе multi-restart.

## Тупиковые направления и извлечённые уроки

### 1. Simulated Annealing
**Что:** Классический SA с перестановками порядка коробов.
**Почему не сработало:** SA требует тысяч итераций для конвергенции. При 900 мс бюджете успевает сделать слишком мало шагов. LNS оказался эффективнее — целенаправленно разрушает и ремонтирует.

### 2. Академический метод (Dell'Amico)
**Что:** Послойная матэвристика + ML-ранжирование из свежей статьи.
**Почему не сработало:** Метод предполагает offline обучение на большом датасете. Для соревнования с жёстким бюджетом по времени и без обучающих данных — слишком тяжёл. Но идея hard block для fragile была ценной.

### 3. Параллельное выполнение стратегий
**Что:** ProcessPoolExecutor для параллельного запуска N стратегий.
**Почему частично не сработало:** Fork overhead на macOS (и в Docker CI) съедал больше времени, чем экономил на параллельности. Для коротких стратегий (10-50 мс) overhead доминирует. Решение: sequential по умолчанию.

### 4. Адаптивное добавление fragile-strict стратегий
**Что:** При fragile_ratio > 0.25 prepend fragile_strict стратегий в начало списка.
**Почему не сработало:** Вытеснило core-стратегии из бюджета n_restarts=10. random_mixed имеет fragile_ratio=0.589, всегда триггерило adaptive head, и скор упал (0.7483 → 0.7266).

### 5. Score-based acceptance в LNS
**Что:** Принимать новое решение, если proxy_score лучше (а не просто placed_count).
**Почему не сработало:** Score-based acceptance принимал решения с меньшим количеством коробов, но "лучшим" скором (из-за отсутствия fragile violations). Результат — потеря покрытия на mixed_column_repeat (77 → 75 placed).

### 6. Синхронизация внутренних эвристик с валидатором
**Что:** Изменить _fast_score и scoring.py, чтобы fragile-on-fragile не считалось нарушением (как в валидаторе).
**Почему не сработало:** Строгий внутренний скоринг производил лучшую укладку. Ослабление привело к регрессии fragile_tower (0.7529 → 0.7033), потому что _fast_score стал выбирать другую "лучшую" стратегию.

## Ключевые принципы, выработанные в процессе

1. **Валидатор ≠ внутренние эвристики.** Валидатор должен точно соответствовать спецификации. Внутренние эвристики могут быть строже, если это улучшает качество.

2. **Многостратегийный портфель > одна умная стратегия.** Ни одна стратегия не доминирует на всех сценариях. Multi-restart с разнообразным портфелем — самый надёжный подход.

3. **Контроль временного бюджета — критичен.** time_score = 1.0 при < 1 сек. Любая оптимизация бессмысленна, если солвер выходит за бюджет.

4. **Каждое изменение — бенчмарк.** Интуиция обманчива. "Улучшение" легко вызывает регрессию на другом сценарии. CI бенчмарк после каждого коммита — обязательно.

5. **Простые фиксы > сложные алгоритмы.** Исправление ошибки в валидаторе (+0.04) дало больше, чем недели оптимизации алгоритмов (+0.06 за 16 итераций).

6. **Постобработка — дешёвый буст.** Compact + insert + reorder дают стабильные +1-3 коробки почти бесплатно по времени.
