## 1. Постановка задачи

Задача относится к классу **3D Bin Packing Problem (NP-hard)** для соревнования X5 Tech AI Championship. Необходимо уложить набор коробов (различных размеров, весов, с флагами хрупкости, ориентации и стекируемости) в паллету ограниченных размеров, максимизируя интегральный скор.

### Входные данные

- **Паллета**: длина (L), ширина (W), максимальная высота (H), максимальный вес (max_weight_kg)
- **Короба**: для каждого SKU — три измерения (length, width, height в мм), вес (кг), количество экземпляров, а также булевы флаги:
  - `strict_upright` — запрет вращения вокруг X и Y осей (только поворот в плоскости пола)
  - `fragile` — нельзя ставить под не-хрупкий груз тяжелее 2 кг
  - `stackable` — если `false`, ничего нельзя ставить поверх

### Жёсткие ограничения (нарушение = 0 баллов)

1. Все короба внутри границ паллеты
2. Нет пересечений (AABB-коллизий) между коробами
3. Каждый короб имеет >= 60% опоры снизу (пол или другие коробы)
4. Соблюдение `strict_upright` — высота коробa должна быть по оси Z
5. Суммарный вес <= max_weight_kg
6. На `stackable=false` короба ничего нельзя ставить

### Формула финального скора

```
final_score = 0.50 * volume_utilization
            + 0.30 * item_coverage
            + 0.10 * fragility_score
            + 0.10 * time_score
```

Где:
- **volume_utilization** = сумма объёмов размещённых коробов / объём паллеты
- **item_coverage** = количество размещённых экземпляров / общее количество запрошенных
- **fragility_score** = max(0, 1 - 0.05 * кол-во нарушений хрупкости). Нарушение — размещение **не-хрупкого** груза тяжелее 2 кг на хрупком коробе. Хрупкий на хрупком — допустимо.
- **time_score**: <= 1 сек = 1.0; 1-5 сек = 0.7; 5-30 сек = 0.3; > 30 сек = 0.0

## 2. Архитектура решения

Решение имеет конвейерную (pipeline) архитектуру из 4 основных этапов:

```
┌──────────────────┐    ┌──────────────┐    ┌──────────────────┐    ┌──────────────────┐
│   Multi-Restart  │───>│     LNS      │───>│  Post-processing │───>│  Выбор лучшего   │
│   (N стратегий)  │    │ (destroy +   │    │  (compact +      │    │  (proxy score)   │
│                  │    │  repair)     │    │   reorder +      │    │                  │
│                  │    │              │    │   insert +       │    │                  │
│                  │    │              │    │   refill)        │    │                  │
└──────────────────┘    └──────────────┘    └──────────────────┘    └──────────────────┘
       ~40%                  ~25%                 ~35%               бюджета времени
```

Каждый этап работает в рамках жёсткого временного бюджета, разделённого на доли от общего лимита (по умолчанию 900 мс для гарантии time_score = 1.0).

## 3. Представление данных

### Модели

- **Box** — тип SKU с полями: sku_id, три измерения, вес, количество, флаги (strict_upright, fragile, stackable). Вычисляемые свойства: volume, base_area.
- **Pallet** — параметры паллеты (длина, ширина, макс. высота, макс. вес). Вычисляемое свойство: volume.
- **Placement** — конкретное размещение экземпляра в паллете: позиция (x, y, z), ориентированные размеры (length, width, height по осям X, Y, Z), rotation_code. Вычисляемые свойства: x_max, y_max, z_max, volume, base_area.
- **Solution** — набор Placement + список неразмещённых + время решения + версия солвера.

### Разворачивание SKU

Перед упаковкой каждый SKU с quantity > 1 разворачивается в список пар `(Box, instance_index)`. Например, SKU с quantity=5 порождает 5 индивидуальных экземпляров для размещения.

## 4. Генерация ориентаций

Каждый короб может быть повёрнут 6 способами — все перестановки трёх измерений по осям X, Y, Z:

| Код   | X-ось   | Y-ось   | Z-ось   |
|-------|---------|---------|---------|
| LWH   | Length  | Width   | Height  |
| LHW   | Length  | Height  | Width   |
| WLH   | Width   | Length  | Height  |
| WHL   | Width   | Height  | Length  |
| HLW   | Height  | Length  | Width   |
| HWL   | Height  | Width   | Length  |

**Фильтрация**:
- Если `strict_upright = true`, допускаются только ориентации, где оригинальная высота (H) направлена по оси Z (коды LWH и WLH).
- Дублирующиеся размеры отсеиваются (например, если L = W, ориентации LWH и WLH дадут одинаковые физические размеры — берётся только одна).

## 5. Extreme Points — генерация кандидатных позиций

Метод **Extreme Points (EP)** — ключевая эвристика для генерации позиций-кандидатов, куда можно разместить следующий короб. Это альтернатива полному перебору координат сетки.

### Инициализация

Начальный набор EP = {(0, 0, 0)} — левый нижний дальний угол паллеты.

### Обновление при размещении нового короба

При размещении короба с координатами (x_min, y_min, z_min) — (x_max, y_max, z_max) генерируются новые EP:

1. **Правая грань**: (x_max, y_min, z_min)
2. **Передняя грань**: (x_min, y_max, z_min)
3. **Верхняя грань**: (x_min, y_min, z_max), (x_max, y_min, z_max), (x_min, y_max, z_max)
4. **Проекции на существующие коробы**: пересечения рёбер нового короба с гранями существующих

### Фильтрация

- Удаляются EP, оказавшиеся внутри только что размещённого короба
- Проверка на попадание внутрь любого существующего короба (через пространственный индекс)
- Проверка границ паллеты

### Сортировка и ограничение

EP сортируются по приоритету **(z, x, y)** — предпочтение отдаётся нижним позициям (заполнение снизу вверх). Количество EP ограничено **200 позициями** для контроля вычислительной сложности.

## 6. Управление состоянием паллеты (PalletState)

PalletState хранит мутабельное состояние паллеты в процессе упаковки:

### Структуры данных

- **boxes** — список размещённых PlacedBox (AABB + метаданные)
- **current_weight** — текущий суммарный вес
- **max_z** — максимальная высота укладки
- **extreme_points** — список кандидатных позиций
- **boxes_by_top_z** — словарь {z_max → [PlacedBox]}, индекс по верхней грани для быстрой проверки опоры
- **xy_index** — пространственный хеш с ячейкой 200 мм для быстрого поиска коллизий

### Проверка can_place()

Метод последовательно проверяет все жёсткие ограничения:

1. **Границы**: координаты размещения в пределах паллеты
2. **Вес**: текущий_вес + вес_нового <= max_weight_kg
3. **Коллизии**: AABB-проверка через пространственный индекс (z проверяется первым для раннего выхода)
4. **Опора**: на z = 0 — пол; иначе вычисляется площадь перекрытия с коробами, чей z_max == z размещения. Требуется >= 60%. Дополнительно: если опорный короб имеет stackable = false, размещение запрещено.

### Вспомогательные методы

- **contact_area_with_neighbors** — суммарная площадь контакта с соседними коробами и стенками паллеты (для scoring)
- **get_fragile_boxes_at_top** — поиск хрупких коробов с верхней гранью на заданной высоте z, перекрывающихся по XY

## 7. Функция оценки размещения (Scoring)

Каждый кандидат (EP × ориентация) оценивается функцией из 5 компонент:

```
score = W_h * height_score
      + W_c * contact_score
      + W_f * fragility_component
      + W_fi * fill_score
      + W_l * layer_score
```

### Компоненты

| Компонента          | Формула                                    | Смысл                                         |
|--------------------|--------------------------------------------|-----------------------------------------------|
| height_score       | 1 - new_max_z / max_height                  | Ниже = лучше (минимизация роста высоты)        |
| contact_score      | min(1, contact_area / surface_area)          | Контакт со стенками и соседями (стабильность)  |
| fragility_component| 1.0 или 0.0 (штраф) или -1.0 (жёсткий блок) | Штраф за размещение тяжёлого на хрупком        |
| fill_score         | 1 - z / max_height                           | Заполнение снизу вверх (заполнение пустот)     |
| layer_score        | 0.0 / 0.5 / 1.0                             | Выравнивание по слоям (плоская укладка)        |

### Layer Score подробнее

- 1.0, если это первый короб (на полу)
- 0.5, если z_min короба совпадает с z_max какого-либо существующего короба (начало нового слоя)
- 1.0, если z_max нового короба совпадает (в пределах 5 мм) с z_max существующего короба (завершение плоского слоя)

### Профили весов

Решение использует несколько профилей весов для диверсификации стратегий:

| Профиль        | height | contact | fragility | fill | layer |
|----------------|--------|---------|-----------|------|-------|
| default        | 0.30   | 0.25    | 0.10      | 0.20 | 0.15  |
| contact_heavy  | 0.25   | 0.30    | 0.10      | 0.20 | 0.15  |
| fill_heavy     | 0.20   | 0.25    | 0.10      | 0.30 | 0.15  |
| layer_heavy    | 0.20   | 0.20    | 0.10      | 0.20 | 0.30  |
| fragile_avoid  | 0.25   | 0.20    | 0.25      | 0.15 | 0.15  |
| compact        | 0.30   | 0.25    | 0.05      | 0.25 | 0.15  |
| wall_hugger    | 0.20   | 0.35    | 0.10      | 0.15 | 0.20  |
| fragile_strict | 0.15   | 0.20    | 0.40      | 0.15 | 0.10  |

Профиль `fragile_strict` жёстко блокирует размещение (возвращает -1.0), если тяжёлый груз (> 2 кг) оказывается на хрупком.

## 8. Алгоритмы упаковки (Packers)

### 8.1 Greedy Packer (основной)

Жадный алгоритм — основа решения:

1. Отсортировать коробы по заданному ключу сортировки
2. Для каждого экземпляра:
   a. Перебрать все EP × допустимые ориентации
   b. Для каждого кандидата: проверить can_place(), вычислить score
   c. Выбрать кандидата с максимальным score
   d. Разместить (place) или добавить в unplaced

Оптимизации:
- Кеширование ориентаций по SKU
- Периодическая проверка временного лимита (каждые ~5% коробов)
- Не копируем список EP при переборе (EP не модифицируются во время scoring)

### 8.2 Two-Phase Packer

Двухфазная упаковка для сценариев со смешанной хрупкостью:

1. **Фаза 1** — не-хрупкие короба (формируют стабильную базу)
2. **Фаза 2** — тяжёлые хрупкие (weight > 2 кг)
3. **Фаза 3** — лёгкие хрупкие (на самый верх, под ничем тяжёлым)

Каждая фаза использует тот же жадный алгоритм EP + scoring, но разделение обеспечивает, что хрупкие коробы не оказываются под тяжёлыми.

### 8.3 Layer Packer

Слоёвый упаковщик: тот же EP + scoring, но с дополнительным бонусом +0.15 для кандидатов, размещаемых на текущем z-уровне (определяется как z_min, который чаще всего встречается среди уже размещённых коробов). Это стимулирует послойное заполнение.

### 8.4 Beam Search Packer

Луч поиска (beam search) — хранит top-K частичных решений параллельно:

1. Инициализация: beam = [(пустое состояние)]
2. Для каждого очередного короба:
   a. Для каждого варианта в beam: перебрать все EP × ориентации
   b. Собрать все кандидаты, отсортировать по score
   c. Оставить top-K лучших вариантов (обрезка beam)
3. Из финального beam выбрать вариант с максимальным количеством размещённых коробов

Параметры: beam_width = 4, time_limit = 500 мс. Требует deepcopy состояния, поэтому медленнее жадного.

### 8.5 Best-Fit Packer (для малых экземпляров)

Best-first жадный: на каждом шаге выбирает лучшую комбинацию (короб, позиция, ориентация) из ВСЕХ оставшихся коробов. Сложность O(N^2 * |EP| * |orientations|), поэтому используется только при <= 50 экземплярах.

### 8.6 Ключи сортировки

Решение включает 15+ именованных ключей сортировки плюс 10 псевдослучайных:

| Ключ                    | Логика                                                   |
|------------------------|----------------------------------------------------------|
| constrained_first      | Ограниченные (upright, non-stackable) первыми, затем объём |
| base_area_desc         | По убыванию площади основания                              |
| fragile_last           | Не-хрупкие первыми, хрупкие последними                     |
| volume_desc            | По убыванию объёма                                         |
| volume_asc             | По возрастанию объёма                                      |
| density_desc           | По убыванию плотности (вес/объём)                          |
| non_stackable_last     | Non-stackable последними                                  |
| height_desc            | По убыванию высоты                                        |
| weight_desc            | По убыванию веса                                          |
| max_dim_desc           | По убыванию наибольшего измерения                          |
| heavy_base_fragile_top | Тяжёлые не-хрупкие → тяжёлые хрупкие → лёгкие хрупкие     |
| stackable_base         | Stackable не-хрупкие → хрупкие → non-stackable            |
| score_per_kg           | По убыванию объём/вес (marginal score per kg)              |
| light_fillers_first    | Лёгкие первыми (максимизация coverage при лимите веса)     |
| coverage_optimal       | По возрастанию вес * объём                                |
| random_0..9            | Детерминированные псевдослучайные перестановки (через MD5) |

## 9. Multi-Restart оркестрация

Центральный компонент, координирующий запуск стратегий.

### Формирование списка стратегий

Стратегия = тройка **(ключ сортировки, профиль весов, тип пакера)**. Список из 40+ стратегий формируется с учётом:

1. **Адаптивная голова** — при высоком давлении по весу (weight_pressure > 1.3) добавляются стратегии score_per_kg и light_fillers_first вперёд списка
2. **Ядро** (позиции 0-11) — наиболее эффективные комбинации, покрывающие основные сценарии
3. **Fragile-strict блок** (позиции 12-14) — для жёсткого предотвращения нарушений хрупкости
4. **Расширенные стратегии** — two-phase, layer, дополнительные профили
5. **Random sorts** — для экплорации пространства решений

### Выбор лучшего

Каждая стратегия оценивается **proxy-скором** (_fast_score) — облегчённая реплика формулы валидатора, считающая:

```
proxy = 0.50 * vol_util + 0.30 * coverage + 0.10 * fragility + 0.10 * 1.0
```

Предполагает time_score = 1.0 (время будет определено в конце), и считает все heavy-on-fragile нарушения (включая fragile-on-fragile) — это осознанное решение: строгий внутренний скоринг производит более качественную укладку, чем правильный по спецификации.

### Адаптивная остановка

Если после нескольких стратегий оценочное время следующей стратегии превышает оставшийся бюджет (40% от 900 мс), итерации прекращаются.

### Fragile-safe кандидат

Параллельно с основным лучшим решением отслеживается лучшее решение с нулём нарушений хрупкости. В конце, если fragile-safe кандидат превышает основного по proxy score, он выбирается вместо основного.

## 10. LNS — Large Neighborhood Search

LNS применяется как метаэвристика для улучшения найденного решения через цикл разрушения-восстановления (destroy-repair).

### Алгоритм

```
Вход: initial_solution, параметры
best = initial_solution
for iteration in 0..max_iterations:
    1. DESTROY: удалить ~30% размещений из best
    2. Перестроить PalletState из оставшихся (валидировать опору!)
    3. REPAIR: жадно переразместить удалённые + неразмещённые + "сироты"
    4. Если new_placed_count > best_placed_count
       или (equal_count и new_volume > best_volume):
         best = new_solution
```

### Стратегии разрушения (чередуются по iteration % 3)

1. **Случайное** — равномерная случайная выборка
2. **Worst-position** — удаление коробов с наибольшим z и наименьшим объёмом (badness = z * 1000 - volume)
3. **Fragility-aware** — целевое удаление коробов, участвующих в нарушениях хрупкости

### Восстановление состояния

При rebuild из оставшихся размещений каждое проверяется через can_place(). Если короб потерял опору (из-за удаления коробов ниже), он становится "сиротой" и добавляется в repair pool с приоритетом.

### Repair pool

Порядок: сироты (приоритет) → удалённые → изначально неразмещённые. Не-сиротская часть перемешивается для разнообразия.

### Диверсификация профилей

Профили весов чередуются по итерациям: default → contact_heavy → fill_heavy.

### Критерий принятия

Строго монотонный: принимается, только если количество размещённых коробов больше, или равно, но суммарный объём больше. Score-based acceptance тестировался, но вызывал регрессии (принимал решения с меньшим покрытием).

## 11. Постобработка (Post-processing)

6-этапный конвейер, применяемый к лучшему решению после LNS.

### Этап 1: Compact Downward

Опускание коробов вниз для заполнения вертикальных зазоров:

1. Обработка коробов сверху вниз (по убыванию z)
2. Для каждого: найти минимальный z, где есть полная опора
3. Проверить: нет коллизий, опора >= 60%, не нарушен stackable
4. Если валидно — опустить

### Этап 2: Reorder Fragile (первый проход)

Уменьшение нарушений хрупкости тремя стратегиями:

**Стратегия 1 — Lift Fragile to Top:**
- Найти хрупкие коробы с тяжёлым грузом сверху
- Удалить их из укладки
- Переразместить на максимальный z (на самый верх), где нет тяжёлого сверху

**Стратегия 2 — Full Repack:**
- Удалить ОБА участника нарушения (тяжёлый + хрупкий)
- Переразместить: сначала тяжёлые не-хрупкие, потом тяжёлые хрупкие, потом лёгкие хрупкие
- Принять, только если нарушений стало меньше и ни один короб не потерян

**Стратегия 3 — Heavy-only Repack:**
- Удалить только тяжёлые коробы-нарушители
- Переразместить с штрафом -0.5 за размещение на хрупком

### Этап 3: Try Insert Unplaced

Попытка вставить неразмещённые коробы в оставшиеся пустоты:

1. Пересчитать PalletState из текущей укладки
2. Сгенерировать дополнительные EP сканированием границ коробов (gap EPs) — сетка из координат x_coords × y_coords × z_tops
3. Объединить стандартные EP + gap EPs
4. Для каждого неразмещённого: найти лучшую позицию с штрафом -0.3 за тяжёлый-на-хрупком
5. После каждой вставки — регенерировать gap EPs
6. **Grid search fallback**: для <= 3 оставшихся неразмещённых — полный перебор по сетке с шагом 50 мм, затем 20 мм

**Порядок вставки**: non-fragile stackable → heavy non-fragile → heavy fragile → light fragile → non-stackable (стабильная база → хрупкий верх).

### Этап 4: Reorder Fragile (второй проход)

Повторное применение reorder_fragile, т.к. вставка на этапе 3 могла создать новые нарушения.

### Этап 5: Remove and Refill

Удаление "плохо" размещённых коробов и попытка переразместить:

1. Оценка "плохости": badness = z * 1000 - volume (высоко и маленький = плохо)
2. Удалить ~15% худших размещений
3. Добавить изначально неразмещённые
4. Переразместить жадно с fill_heavy профилем и штрафом за тяжёлый-на-хрупком
5. Принять, если размещено >= столько же коробов и нарушений хрупкости не больше

### Этап 6: Final Compact

Повторный compact_downward, если были вставлены новые коробы. С проверкой через _validate_placement_feasibility.

## 12. Валидатор и формула финального скора

Валидатор (validator.py) выполняет полную проверку решения, идентичную серверной:

### Жёсткие проверки (в порядке)

1. Anti-cheat: размеры размещённого короба — перестановка исходных
2. Валидность rotation_code (одно из 6 значений)
3. Количество экземпляров SKU <= заявленного quantity
4. Перевес: total_weight <= max_weight_kg
5. Границы паллеты
6. Strict upright: высота размещённого == оригинальная высота
7. AABB-коллизии
8. Гравитация: опора >= 60%
9. Stackable: ничего на non-stackable

### Подсчёт Fragility Score

```
violations = 0
for каждого размещённого top:
    if top.weight <= 2.0: skip
    if top.fragile: skip        # Хрупкий на хрупком — НЕ нарушение (по спецификации)
    for каждого bottom с fragile=true:
        if top стоит на bottom (z_min == z_max, XY overlap > 0):
            violations += 1

fragility_score = max(0, 1 - 0.05 * violations)
```

### Ключевое проектное решение

Валидатор считает нарушения **по спецификации** (fragile-on-fragile допустимо), а внутренние эвристики солвера (_fast_score, scoring.py, postprocess) используют **строгий подсчёт** (ВСЕ heavy-on-fragile, включая fragile-on-fragile). Это осознанный выбор: строгие внутренние критерии производят более качественную укладку, т.к. стимулируют избегать размещения чего-либо на хрупких коробах.

## 13. Управление временным бюджетом

### Общий бюджет

Внутренний лимит жёстко ограничен **900 мс** (вне зависимости от входного параметра), чтобы гарантировать time_score = 1.0 (< 1 сек).

### Распределение бюджета

| Фаза           | Доля    | Назначение                                     |
|----------------|---------|-------------------------------------------------|
| Стратегии      | ~40%    | Multi-restart с адаптивной остановкой           |
| LNS            | ~25%    | Destroy-repair (если осталось > 400 мс)         |
| Постобработка  | ~35%    | Compact + reorder + insert + refill + compact   |
| Best-fit       | -       | Если < 50 items и < 50% бюджета прошло          |

### Адаптивные механизмы

- **Adaptive stop** в стратегиях: отслеживается максимальное время стратегии, прогнозируется, влезет ли следующая
- **LNS skip**: если осталось < 400 мс, LNS пропускается
- **Postprocess budget cascade**: каждый этап получает долю оставшегося времени, с проверками перед каждым

## 14. Ключевые проектные решения и уроки

### 1. Разделение валидатора и внутренних эвристик

Валидатор (формула оценки) должен точно соответствовать серверным правилам. Но внутренние эвристики солвера (scoring, proxy score, postprocess) могут использовать более строгие критерии, если это улучшает качество укладки. Пример: считать ВСЕ heavy-on-fragile (включая fragile-on-fragile) как нарушения внутри солвера, хотя валидатор их не штрафует.

**Урок**: Попытка синхронизировать внутреннюю логику с валидатором привела к регрессии fragile_tower (0.7529 → 0.7033), потому что _fast_score стал выбирать другую "лучшую" стратегию.

### 2. Стабильность acceptance criteria в LNS

Score-based acceptance (принимать, если score больше) приводило к потере коробов: решение с меньшим количеством коробов, но лучшим score из-за отсутствия нарушений, заменяло решение с большим покрытием. Монотонный критерий (больше коробов → лучше) оказался надёжнее.

### 3. Adaptive portfolio — двойной меч

Адаптивное добавление fragile_strict стратегий в начало списка при высокой доле хрупких товаров (fragile_ratio > 0.25) вытеснило эффективные ядерные стратегии из бюджета n_restarts=10, вызвав регрессию random_mixed. Решение: убрать адаптивное добавление, вместо этого иметь fragile_strict стратегии на фиксированных позициях в ядре.

### 4. EP cap = 200

200 EP — баланс между качеством (больше кандидатов) и скоростью. Увеличение до 500 не давало значимого улучшения, но замедляло каждую стратегию.

### 5. Пространственная индексация

Spatial hash (ячейка 200 мм) + boxes_by_top_z — критически важны для производительности. Без них проверка can_place() была бы O(N) по всем коробам; с индексами — O(k), где k — количество коробов в соседних ячейках.
